== 📖 Sobre

*NostrTube* é uma plataforma descentralizada de compartilhamento de vídeos, construída como um aplicativo de página única em React. O sistema utiliza o protocolo Nostr para recursos sociais e distribuição de metadados, enquanto usa servidores Blossom para armazenamento descentralizado de vídeos. Essa arquitetura permite a hospedagem de vídeos resistentes à censura sem depender de infraestrutura centralizada.

== Arquitetura de Eventos de Vídeo

O domínio de vídeos separa explicitamente os tipos de conteúdo publicados em Nostr:

* `34235` e `21`: vídeos normais, horizontais ou longos.
* `34236` e `22`: vídeos curtos, verticais ou de consumo no formato Shorts.

As listagens principais de vídeo usam apenas os kinds normais. A resolução de detalhe por `/v/$eventId`, importação de eventos e análises de interação reconhecem a união dos kinds normais e shorts para preservar compatibilidade com eventos existentes.

=== Classificação de Publicação

O fluxo de publicação aceita uma seleção manual de tipo de conteúdo:

* `auto`: padrão conservador.
* `normal`: força publicação como `34235`.
* `short`: força publicação como `34236`.

No modo `auto`, um vídeo só é publicado como short quando há metadados confiáveis de dimensão e duração:

* altura maior que largura;
* razão `largura / altura` menor ou igual a `0.75`;
* duração menor ou igual a 180 segundos.

Casos ambíguos, importações sem dimensão confiável e URLs externas sem metadados completos permanecem como vídeo normal, a menos que o usuário escolha manualmente `short`.

=== Organização Técnica

As constantes e helpers ficam centralizados em `src/features/video/services/video-kinds.ts`.

Use:

* `NORMAL_VIDEO_EVENT_KINDS` para feeds, buscas e listagens de vídeos normais.
* `SHORT_VIDEO_EVENT_KINDS` para a tela de Shorts.
* `ALL_VIDEO_EVENT_KINDS` para resolução de detalhe, importação, métricas e compatibilidade.

O builder NIP-71 em `src/features/upload/services/video-event-builder.service.ts` permanece como ponto único de montagem de tags e resolve o kind antes de publicar, sem duplicar lógica para shorts.

== Tela de Shorts

A rota `/shorts` renderiza um feed vertical dedicado a eventos de short video (`34236` e `22`). A rota é registrada manualmente em `src/router.ts` e aparece na navegação principal como entrada independente do feed de vídeos normais.

A experiência visual foi ajustada: `/shorts` mantém integração com `AppShell`, sidebar e busca do projeto, mas apresenta uma grade de cards verticais 9:16 para descoberta. A entrada direta `/short/$eventId` abre a experiência fullscreen com navegação vertical um-a-um, preservando `naddr`, `nevent`, `note`, ids hexadecimais e `d` tags sem redirecionar `naddr` para um identificador ambíguo.

O módulo fica em `src/features/shorts/` e segue a separação entre:

* `services/`: filtros NDK e adaptação de eventos para view model.
* `hooks/`: conexão com NDK/NDK Hooks e filtros de visibilidade.
* `components/`: layout vertical, overlay e player dedicado.

A tela `/shorts` usa uma grade responsiva para descoberta e paginação incremental. O player vertical com snap, um item por viewport e controle de reprodução pelo item ativo fica restrito a `/short/$eventId`, onde o usuário já escolheu um short para assistir.

=== Busca de Shorts

A busca da tela de Shorts é isolada da rota `/search` e usa o parâmetro `/shorts?search=...`. O input atualiza o filtro de relays com debounce via `@tanstack/react-pacer`, evitando consultas a cada tecla digitada. O service de Shorts continua montando filtros apenas com `SHORT_VIDEO_EVENT_KINDS`, então a busca não mistura vídeos normais no feed vertical.

=== Paginação, Virtualização e Preload

O feed de Shorts usa `useInfiniteQuery` com filtros NDK independentes do feed principal. Cada página consulta apenas `SHORT_VIDEO_EVENT_KINDS`, com `limit` próprio e cursor `until` calculado a partir do `created_at` do último evento da página anterior.

A renderização imersiva de `/short/$eventId` usa `@tanstack/react-virtual` para manter poucos itens montados por vez, preservando scroll vertical com snap e um vídeo por viewport. O carregamento incremental é acionado ao aproximar o índice ativo do fim da lista e é limitado por `@tanstack/react-pacer` para evitar chamadas sucessivas durante scroll rápido. Em `/shorts`, a paginação usa um sentinel no fim da grade.

O item ativo recebe autoplay e os demais pausam automaticamente. O próximo item virtualizado recebe `preload="auto"`, enquanto os demais permanecem em `metadata`, reduzindo buffering na transição sem pré-carregar todo o feed.

Na rota direta `/short/$eventId`, o evento inicial é injetado no feed antes da paginação para garantir abertura imediata do short solicitado. O parâmetro opcional `author` restringe as próximas páginas aos shorts do autor quando fornecido. A URL só é substituída quando o usuário navega para outro short no deck, preservando a URL inicial compartilhada.

== 🚀 Instalação e Desenvolvimento

. Clone o repositório:
+
[source,bash]
----
git clone https://github.com/gabrielmoura/NostrTube.git
cd NostrTube
----

. Instale as dependências:
+
[source,bash]
----
pnpm install
----

. Execute o servidor de desenvolvimento:
+
[source,bash]
----
pnpm dev
----
+
A aplicação estará disponível em `http://localhost:5173`.

== 🐳 Docker

Build da imagem:

[source,bash]
----
docker build -t nostrtube .
----

Executar o container:

[source,bash]
----
docker run -p 3000:80 nostrtube
----

=== Proxy Image

Para configurar o proxy de imagens necessário:

[source,bash]
----
docker run -d -e="IMGPROXY_ALLOWED_SOURCES:*,https://*,https://*/*,s3://*" -e="IMGPROXY_ALLOW_LOOPBACK_SOURCE_ADDRESSES:true" -e"IMGPROXY_ALLOW_LINK_LOCAL_SOURCE_ADDRESSES:true" -e"IMGPROXY_IGNORE_SSL_VERIFICATION:true" --add-host=host.docker.internal:host-gateway -p=9095:8080 --name="imgproxy"  ghcr.io/imgproxy/imgproxy:latest
----

Configure a variável de ambiente:

[source,env]
----
VITE_APP_IMGPROXY=http://localhost:9095
----
